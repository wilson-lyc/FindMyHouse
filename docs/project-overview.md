# 项目总体设计

## 项目定位

FindMyHouse 是一个帮助用户做租房决策的房源整理与智能推荐系统。它不是单纯的房源发布平台，而是面向个人或小团队的“租房决策工作台”：用户可以把来自不同渠道的房源统一录入、清洗、标注、比较，并通过地图、通勤、预算、生活便利性和 Agent 分析获得更清晰的选择建议。

## 核心问题

租房决策通常面临以下痛点：

- 房源信息分散在不同平台、聊天记录、表格和网页保存记录中。
- 房源价格、面积、位置、押金、可入住时间等字段不统一，难以横向比较。
- “离公司近”不是一个单一指标，还涉及通勤方式、时间、成本、换乘、步行距离和天气影响。
- 用户容易被单个亮点吸引，忽略预算压力、生活便利性、合同风险或长期体验。
- 多人共同租房时，偏好不同，沟通成本较高。

## 产品目标

- 帮助用户把房源数据结构化，形成可持续维护的个人房源库。
- 基于地图展示房源位置、工作地点、通勤路线和周边设施。
- 提供多维度筛选、排序和对比能力。
- 用 Agent 辅助总结房源优劣、发现风险、生成推荐理由。
- 支持从“初筛”到“看房后决策”的完整租房流程。

## 目标用户

- 正在找房的个人用户。
- 需要比较多个工作地点或学习地点通勤情况的用户。
- 夫妻、室友、小团队等多人共同找房场景。
- 希望用数据方式管理租房选择，而不是依赖聊天记录和临时表格的用户。

## 核心使用流程

1. 用户添加工作地点、学校或常去地点。
2. 用户录入房源，可以手动填写、粘贴链接、导入表格，后续可扩展网页解析。
3. 系统对房源地址进行地理编码，生成地图坐标。
4. 系统计算房源到关键地点的距离和通勤方案。
5. 用户通过地图、列表、筛选器和对比视图查看候选房源。
6. Agent 根据用户偏好、预算、通勤权重和房源数据生成分析报告。
7. 用户记录看房反馈、调整偏好，系统更新推荐排序。

## 设计原则

- 数据优先：所有推荐和分析都应尽量基于可解释的结构化字段。
- 空间与管理并重：位置关系是租房决策的核心，列表和表格则保证录入、筛选、比较的效率。
- 可解释推荐：Agent 给出建议时必须说明依据，而不是只给结论。
- 渐进增强：先支持手动录入和本地数据，再逐步扩展自动导入、插件和多用户协作。
- 本地友好：SQLite 适合早期轻量部署和个人数据管理，后续可迁移到 PostgreSQL。
