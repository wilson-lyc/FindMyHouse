## 1. 测试与质量保障

### 1.1 现状

项目**没有任何测试**——无单元测试、无集成测试、无 E2E 测试。唯一的质量保障是 TypeScript 编译时类型检查（`vue-tsc` + `tsc --noEmit`）。

### 1.2 改进建议

| # | 改进项 | 优先级 | 工作量 | 说明 |
|---|--------|--------|--------|------|
| 1.1 | **添加后端单元测试** | P0 | 中 | 后端逻辑相对独立（Service + Repository 模式），使用 vitest 对 HouseService / LocationService / AgentService 的核心路径编写测试，覆盖 CRUD 正常流程和边界情况（如删除不存在的记录）。Repository 层可通过 SQLite `:memory:` 数据库测试。 |
| 1.2 | **添加前端组件测试** | P1 | 中 | 对关键组件（HouseFormDialog、MapPanel）使用 vitest + @vue/test-utils 编写组件渲染和交互测试。 |
| 1.3 | **添加 API 集成测试** | P1 | 大 | 使用 Fastify 的 `inject()` 方法，启动最小化 Fastify 实例，测试所有 `/api/houses`、`/api/locations` 端点的请求/响应契约，覆盖 Zod 校验错误场景。 |
| 1.4 | **添加 AI Agent 回归测试** | P1 | 大 | Agent 的 LangGraph 图逻辑可用模拟 LLM（mock LLM 返回固定 JSON）进行测试，确保工具调用链和状态流转正确。 |
| 1.5 | **引入 CI 测试流水线** | P1 | 小 | 在 GitHub Actions 中配置 `npm test` 和 `npm run typecheck` 在每次 PR 时自动运行。 |

## 2. 工程化与开发体验

### 2.1 现状

- 无 Linter（ESLint / Biome）
- 无代码格式化工具（Prettier / dprint）
- 无 Git Hooks（husky / lint-staged）
- 无提交信息规范（commitlint / Conventional Commits）
- tsconfig 中 `strict` 模式已开启，但缺少 `noUncheckedIndexedAccess` 等更严格的选项
- 项目版本号停留在 `0.1.0`，无变更日志（CHANGELOG）

### 2.2 改进建议

| # | 改进项 | 优先级 | 工作量 | 说明 |
|---|--------|--------|--------|------|
| 2.1 | **引入 Biome 统一 Lint + Format** | P0 | 小 | Biome 比 ESLint + Prettier 更快，零配置即用。替换当前的代码风格真空状态，统一前后端的代码风格。 |
| 2.2 | **配置 Git Hooks** | P1 | 小 | 使用 husky + lint-staged，在 commit 前自动运行 Biome 检查和 TypeScript 类型检查。 |
| 2.3 | **收紧 TypeScript 配置** | P1 | 小 | 在 `tsconfig.json` 中开启 `noUncheckedIndexedAccess`、`exactOptionalPropertyTypes`、`noPropertyAccessFromIndexSignature`，减少运行时 undefined 风险。 |
| 2.4 | **引入 Conventional Commits + CHANGELOG** | P2 | 小 | 使用 commitlint 规范提交信息，配合 standard-version 或 changesets 自动生成变更日志。 |
| 2.5 | **添加开发环境 Docker 化** | P2 | 中 | 提供 `docker-compose.yml` 一键启动前后端，降低新贡献者的环境配置成本。 |

## 3. 架构与代码质量

### 3.1 现状

**后端：**
- 分层清晰：Routes → Service → Repository → SQLite
- 使用 Zod 做输入校验，但错误处理过于简单（ZodError → 400，其余 → 500）
- 依赖注入采用手动构造，无 DI 容器
- 部分模块（如 AgentService）在模块内直接 `new` 依赖，不利于测试

**前端：**
- Vue 3 Composition API + Pinia 状态管理，架构合理
- `MainLayout.vue` 承担了过多的职责（~413 行），包括布局、状态管理、事件协调、弹窗控制、路由分发等，违反了单一职责原则
- 部分组件存在模板内联事件处理逻辑（如 `MainLayout.vue` 的模板通过 `provide/inject` 透传大量回调）
- 语义 HTML 标签使用良好（`<main>`, `<aside>`, `<section>`, `<article>`, `<figure>`）
- 组件的 `defineExpose` 暴露 API 给父组件调用（如 `MapPanel.vue` 暴露 11 个方法），形成隐式耦合

### 3.2 改进建议

| # | 改进项 | 优先级 | 工作量 | 说明 |
|---|--------|--------|--------|------|
| 3.1 | **拆分 MainLayout 的职责** | P0 | 大 | 当前 `MainLayout.vue` 集成了布局、路由、房源 CRUD、地点 CRUD、AI Agent 确认、地图联动等职责。建议拆分为：`AppLayout`（纯布局）+ `HouseManager`（房源操作）+ `LocationManager`（地点操作）+ `AgentConfirmHandler`（AI 确认回调），通过组合式函数（composables）或在 Pinia store 中管理。 |
| 3.2 | **统一后端错误处理** | P1 | 中 | 当前错误处理只有简单的 try-catch。建议实现 Fastify 错误插件，统一格式化 ZodError、NotFoundError、BusinessError 等不同类型异常，确保 API 错误响应结构一致。 |
| 3.3 | **引入 DI 容器提升可测试性** | P2 | 中 | 可使用 awilix 或 tsyringe 替代手动 new 依赖，使 Service 的依赖可被 mock。当前 `AgentService` 直接 `new HouseRepository(db)` 导致难以隔离测试 Agent 逻辑。 |
| 3.4 | **减少 defineExpose 隐式耦合** | P1 | 小 | `MapPanel.vue` 通过 `defineExpose` 暴露 11 个方法给父组件调用。建议通过 emit 事件 + Pinia store 状态驱动，降低组件间的隐式耦合。 |
| 3.5 | **提取常量与魔术值** | P2 | 小 | `MapPanel.vue` 中的 `houseFocusZoom = 16`、`locationFocusZoom = 14`、防抖 `300ms` 等数值建议提取到配置文件或常量文件中。 |
| 3.6 | **使用环境变量管理业务配置** | P2 | 小 | 当前后端 tsconfig target `ES2022` 支持顶层 await，但 `.env` 中只有运行配置，没有业务级配置（如默认地图中心点）。建议将 `[116.397428, 39.90923]`（北京天安门）这种默认值从代码中提取到配置。 |

## 4. 性能优化

### 4.1 现状

- SQLite 已开启 WAL 模式
- 通勤路线已做服务端缓存，避免重复请求高德 API
- 地图视野变化 300ms 防抖
- 主布局 `onMounted` 使用 `Promise.all` 并行加载
- 高德地图 SDK 按需懒加载
- LLM 实例在配置不变时复用

**潜在性能问题：**
- 通勤路线对每个房源**并行**请求后端，但后端对每个房源单独请求高德 API。如果有 20 个房源，会并发 20 个 HTTP 请求到高德
- `houses.value.map(...).join('|')` 的 watch 表达式每次房源变更都重新执行
- 地图大头针每次数据变化时 `clearMap()` 后全量重新渲染，没有使用增量更新
- SQLite 查询未在关键路径上建立索引（如 `houses.status` 用于筛选）

### 4.2 改进建议

| # | 改进项 | 优先级 | 工作量 | 说明 |
|---|--------|--------|--------|------|
| 4.1 | **批量化路线请求** | P1 | 中 | 后端增加批量路线查询接口，前端一次请求多个起点到同一个终点的路线，后端复用缓存并行查询高德，减少前端网络开销。 |
| 4.2 | **添加 SQLite 索引** | P1 | 小 | 为 `houses` 表的 `status`、`rentPrice`、`latitude`/`longitude` 字段添加索引，提升筛选查询性能。 |
| 4.3 | **地图标记增量更新** | P1 | 中 | 当前每次数据变化都 `clearMap()` + 全量重新创建标记。优化为计算新旧标记的 diff，只增删改变化的标记，减少 DOM 操作。 |
| 4.4 | **优化 watch 表达式** | P2 | 小 | `houses.value.map(...).join('|')` 每次房源变更都会重新计算整个字符串。建议使用序列化哈希或深度 watch + 脏检查标记来触发路线加载。 |
| 4.5 | **路线缓存添加 TTL** | P2 | 小 | 当前路线缓存无过期机制。建议为 `map_route_cache` 添加 `expires_at` 字段，超过 TTL（如 7 天）的缓存自动失效，避免数据过时仍被使用。 |

## 5. 安全增强

### 5.1 现状

- 使用参数化查询防止 SQL 注入
- 敏感配置（API Key / 高德 Key）有 `type="password"` 输入
- CORS 配置为 `origin: true`（开发阶段接受所有来源）
- 无 CSP 头
- 无速率限制
- 无认证/授权机制（适用单人本地场景，但 Web UI 暴露在网络上时存在风险）

### 5.2 改进建议

| # | 改进项 | 优先级 | 工作量 | 说明 |
|---|--------|--------|--------|------|
| 5.1 | **添加 CSP 头** | P1 | 小 | Fastify 响应中添加 Content-Security-Policy 头，限制脚本来源，防止 XSS 攻击。特别需要注意高德地图 SDK 的 CDN URL 需要在 `script-src` 中放行。 |
| 5.2 | **添加速率限制** | P2 | 小 | 对 `/api/chat` 端点添加速率限制（如 30 次/分钟），防止 LLM API 被滥用导致费用失控。可使用 `@fastify/rate-limit`。 |
| 5.3 | **高德 Key 前端暴露检查** | P2 | 中 | 当前高德 Key 在前端 `index.html` 中通过 `<script>` 标签加载 JS API 时暴露。建议评估是否使用高德的安全密钥（JSAPI 2.0 支持签名验证）。 |
| 5.4 | **API Key 服务端存储加密** | P2 | 小 | LLM API Key 当前以明文存储在 SQLite `app_config` 表中。建议在存储时使用 `crypto` 模块进行 AES 加密，仅在使用时解密。或使用操作系统密钥链（Keychain）。 |

## 6. 用户体验

### 6.1 现状

- 功能覆盖完善：房源 CRUD、地点管理、通勤计算、AI 对话、首次引导
- 缺少批量操作能力
- 数据导入/导出能力缺失
- 房源详细信息的展示密度较低
- 无收藏/标记功能
- 无数据可视化（如租金分布图、通勤时间对比图）
- 移动端适配缺失

### 6.2 改进建议

| # | 改进项 | 优先级 | 工作量 | 说明 |
|---|--------|--------|--------|------|
| 6.1 | **批量导入/导出房源** | P0 | 中 | 支持从 CSV / 贝壳 Excel 模板批量导入房源，以及将当前房源导出为 CSV。这是找房场景中最常见的需求（在不同平台收集房源后统一录入）。 |
| 6.2 | **房源对比功能** | P1 | 中 | 允许用户在列表中选择 2-4 套房源，在对比视图中并排查看租金、户型、通勤时间、费用等关键信息的对比表格。 |
| 6.3 | **租金与通勤可视化** | P1 | 中 | 在地图上以热力图或气泡图展示租金分布，或以条形图展示各房源到焦点地点的通勤时间排序。可以使用 ECharts 或 Chart.js。 |
| 6.4 | **房源备注增强** | P2 | 小 | 当前备注为纯文本。支持 Markdown 富文本备注，或添加标签（如"近地铁"、"新装修"、"房东直租"）进行筛选。 |
| 6.5 | **看房日程管理** | P2 | 大 | 支持为房源添加看房预约时间、添加看房笔记和评分（1-5 星），并在日历视图中展示看房日程。 |
| 6.6 | **移动端适配** | P2 | 大 | 当前布局针对桌面设计，在移动设备上左侧导航和分栏面板无法正常使用。可考虑使用响应式布局或提供移动端 PWA 版本。 |
| 6.7 | **多通勤目的地支持** | P2 | 中 | 当前仅支持一个焦点地点作为通勤终点。建议支持多个目的地（如"公司"+"学校"），并显示综合通勤时间（如取最大值或总和）。 |
| 6.8 | **费用统计看板** | P2 | 中 | 在仪表盘中汇总所有房源的费用（租金 + 物业费 + 水/电费）和付款周期，帮助用户做预算对比。 |

- 核心功能：房源管理、地点管理、地图通勤、AI 助手
- AI 助手仅支持文本对话，不支持图片分析
- 无数据持久化导出
- 无历史记录版本对比

### 7.2 改进建议

| # | 改进项 | 优先级 | 工作量 | 说明 |
|---|--------|--------|--------|------|
| 7.1 | **房源图片管理** | P1 | 中 | 房源信息中增加图片上传（可拖拽），在详情卡片和信息窗口中展示缩略图。后端可使用本地文件存储或集成对象存储。 |

## 8. 部署与运维

### 8.1 现状

- 仅支持 `npm run dev` 本地开发运行
- 有 `npm run build` 打包，但生产部署文档不完整
- 无 Docker 化
- 无 CI/CD 流水线
- 无日志分级和日志轮转
- 无健康检查端点（现有 `/api/health` 仅返回 `{ status: 'ok' }`，未包含数据库连接检查）

### 8.2 改进建议

| # | 改进项 | 优先级 | 工作量 | 说明 |
|---|--------|--------|--------|------|
| 8.1 | **增强健康检查端点** | P1 | 小 | 当前 `/api/health` 只返回固定字符串。建议检查数据库连接和 LLM 配置状态，返回详细的健康状态 JSON。 |
| 8.2 | **Docker 化部署** | P1 | 中 | 编写 `Dockerfile`（多阶段构建）和 `docker-compose.yml`，包含后端、前端 Nginx 静态文件服务，以及卷挂载 SQLite 数据文件。 |
| 8.3 | **引入日志框架** | P1 | 中 | 当前后端使用 `console.error` 记录错误。建议引入 pino（Fastify 内置）统一日志格式，支持日志级别（info/warn/error）和日志轮转。 |
| 8.4 | **GitHub Actions CI** | P1 | 小 | 配置 CI 流水线：安装依赖 → 类型检查 → Lint → 测试 → 构建。确保每次提交代码质量。 |
| 8.5 | **数据库备份机制** | P2 | 小 | SQLite 数据文件备份策略：定期备份脚本或使用 `VACUUM INTO` 命令，避免数据丢失。 |

## 结论

FindMyHouse 作为个人项目，核心功能已经趋于完整，架构设计合理，代码风格一致。当前的瓶颈主要集中在三个方面：

1. **质量保障缺失**：零测试覆盖，随着功能迭代，回归风险会越来越高
2. **工程化基础薄弱**：缺少 Lint、Format、CI 等现代前端工程必备基础设施
3. **扩展场景受限**：无法批量导入/导出数据，限制了作为找房工具的主要使用场景

建议优先从 **P0 项**入手，投入产出比最高，能在较短时间内显著提升项目可维护性和用户体验。
